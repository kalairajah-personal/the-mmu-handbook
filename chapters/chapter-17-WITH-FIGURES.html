17.1 Introduction: The Hidden Machine Inside the MMU

Every TLB miss sets a hardware machine in motion. That machine — the page table walker (PTW) — traverses a multi-level radix tree in memory, extracts the physical frame number and permission bits from the leaf entry, and installs the result into the TLB. From the software perspective this happens invisibly and automatically. From the microarchitecture perspective, it is one of the most performance-critical circuits on a modern processor.

Chapters 3 and 4 of this book explain what the page table walker does: which bits of the virtual address index each level of the page table tree, how page walk caches (PWC) eliminate redundant upper-level fetches, and what the resulting latency looks like from the software side. This chapter asks a different question: how is the PTW physically implemented in hardware? The answer determines how many TLB misses a processor can service concurrently, whether a speculative walk can observe a kernel memory mapping before the permission check fires, and why the cost of a cold TLB miss can vary by two orders of magnitude depending on where the page table entries reside in the memory hierarchy.

Understanding PTW microarchitecture is no longer a purely academic exercise. As working sets grow beyond the reach of on-chip TLBs — a challenge documented in detail in Chapters 14 and 16 — the PTW becomes a throughput bottleneck rather than an occasional latency spike. And as Chapter 18 will show, the PTW's speculative behaviour creates a category of security vulnerability that is only comprehensible once the underlying hardware mechanism is understood.

This chapter covers the PTW pipeline stages and their latency characteristics, the Miss-Status Holding Register (MSHR) mechanism that enables concurrent walks, the conditions under which a processor initiates a speculative walk, and the design differences between the x86-64 microcode-assisted walker and the ARM64 hardware FSM. The RISC-V software-managed approach is examined as a deliberate architectural contrast, with attention to the implementations that add optional hardware walkers to the base specification.

17.2 PTW Pipeline Stages

A hardware page table walk for a 4-level x86-64 virtual address proceeds through a fixed sequence of stages. Each stage issues a memory read to fetch one level of the page table tree, checks the resulting entry for validity and permissions, and computes the base address for the next level. The sequence is as follows.

Stage 0 — TLB miss detection. The load or instruction fetch unit detects that the virtual address has no matching entry in the TLB. The pipeline stalls the dependent instruction and raises an internal TLB miss signal to the page table walker logic. This detection consumes 1–4 cycles depending on whether the pipeline is in-order or out-of-order and how far the miss propagates before being detected.

Stage 1 — CR3 load. The walker reads CR3 (Control Register 3), which contains the physical base address of the PML4 table for the current address space. On x86-64, CR3 also carries the PCID field in bits [11:0]. The PML4 base address is computed as CR3[51:12] << 12.

Stage 2 — PML4E fetch. The walker uses bits [47:39] of the virtual address as an index into the PML4 table and issues a read to physical address PML4_base + VA[47:39] * 8. The resulting 8-byte entry is checked for the Present bit (P=1); if absent, a page fault (#PF) is queued for delivery. If the PS bit is set, the walk terminates here with a 512 GB huge page.

Stage 3 — PDPTE fetch. Bits [38:30] of the virtual address index the Page Directory Pointer Table. The walker fetches from PDPT_base + VA[38:30] * 8. A PS bit here signals a 1 GB page. An absent Present bit queues a page fault.

Stage 4 — PDE fetch. Bits [29:21] index the Page Directory. A PS bit here signals a 2 MB large page; the walk terminates. This is the most common large page size in production systems — Transparent Huge Pages and GPU mappings both use 2 MB.

Stage 5 — PTE fetch. Bits [20:12] index the Page Table itself, yielding the 4 KB leaf entry. This entry contains the Physical Frame Number in bits [51:12], the Present bit, Read/Write, User/Supervisor, NX, and the Accessed/Dirty bits.

Stage 6 — TLB fill. The walker extracts the PFN and permission bits from the PTE and installs a new TLB entry. The stalled instruction is replayed; the TLB now hits and the load proceeds.

The latency of stages 2–5 depends entirely on where the page table entries reside in the memory hierarchy. Each fetch is an ordinary 8-byte load that traverses the same L1D → L2 → L3 → DRAM hierarchy as any other memory access. An L1D hit resolves in 4–5 cycles; an LLC miss to DRAM adds approximately 200 cycles on a DDR5 system. A cold walk with all four levels missing from cache therefore costs roughly 4 × 200 = 800 cycles in the worst case, or 200–250 ns at 4 GHz. The page walk cache (described in Chapters 3 and 4) eliminates the upper-level fetches when those entries are recently used, reducing a typical warm walk to a single PTE fetch of 12–50 cycles.

The ARM64 Translation Table Walk (TTW) proceeds through the same number of levels for a 48-bit virtual address space: L0 → L1 → L2 → L3, indexed by VA[47:39], VA[38:30], VA[29:21], and VA[20:12] respectively. The base address is held in TTBR0_EL1 (user-space mappings, VA[55]=0) or TTBR1_EL1 (kernel mappings, VA[55]=1). The TTW is implemented as a hardware finite-state machine rather than a microcode-assisted sequence; the practical latency characteristics are similar to x86-64 since both designs ultimately issue loads through the same L1D cache hierarchy.

A key architectural difference is that the ARM64 TTW is guaranteed to be cache-coherent with the data cache: a store to a PTE that enters the L1D cache will be observed by a subsequent translation table walk on the same core. On x86-64, the same coherency guarantee exists but is defined by the paging consistency rules in SDM Vol 3A §4.10.4, discussed in Chapter 20.

17.3 Miss-Status Holding Registers for Page Walks

A cache miss for ordinary data is handled by a Miss-Status Holding Register (MSHR). When a load misses in the L1D cache, the processor allocates an MSHR entry to track the outstanding miss, records the load's destination register, and continues executing independent instructions from the out-of-order window. When the cache line arrives from L2 or DRAM, the MSHR triggers a cache fill and wakes the waiting instruction.

Page table walks require the same mechanism at a higher level. When a TLB miss occurs and the PTW begins a walk, the processor allocates a PTW-MSHR entry to track the outstanding walk. This entry records the virtual address being walked, the current stage of the walk (which page table level has been fetched so far), and the set of load queue entries that are blocked waiting for this TLB entry to be installed.

The critical design decision is how many PTW-MSHR entries the processor provides, because this determines how many TLB misses can be serviced concurrently. Intel processors from the Haswell microarchitecture onward support two simultaneous hardware page table walkers, with two corresponding MSHR slots. The AMD Zen 4 architecture supports up to four concurrent walks per core. ARM Cortex-A710 supports two concurrent TTWs. These numbers set a hard ceiling on PTW throughput: a workload that generates more TLB misses per cycle than there are walker slots will stall in the load queue regardless of memory bandwidth.

Coalescing. A critical MSHR optimisation is coalescing: if two loads miss in the TLB and both reference addresses within the same 4 KB page, they require the same PTE. The processor recognises this — the PTW-MSHR uses the page-aligned virtual address (VA[63:12]) as its tag — and services both loads from a single walk. Load 1 at VA 0x7FFF_A000 and Load 2 at VA 0x7FFF_A010 share a PTE; only one MSHR entry and one walk are consumed. This coalescing behaviour is the well-known observation that accessing N distinct cache lines within the same page costs only one TLB miss, not N.

Blocking behaviour. When all PTW-MSHR slots are occupied and a new TLB miss arrives, the processor cannot initiate a new walk. The offending load stalls in the load queue and the out-of-order window drains. This is the PTW bottleneck identified by Basu et al. [2] as the primary constraint on TLB performance for large working set workloads: the bottleneck is not the latency of any individual walk but the serialisation imposed by a small number of concurrent walker slots.

The interaction between PTW-MSHRs and the page walk cache (detailed in Chapters 3 and 4) is synergistic. A walk that hits in the PWC at the PDPTE level (stage 3) completes in 12–20 cycles rather than 200+, freeing the MSHR slot much sooner. Under a high-miss workload with hot page tables, PWC hits dominate and walker throughput approaches the MSHR allocation rate rather than the DRAM bandwidth limit.

17.4 Speculative Page Table Walks

Modern out-of-order processors do not wait until a load reaches the head of the retirement queue to initiate a TLB lookup. The TLB is accessed speculatively, during the issue stage, on the basis of an address computed from potentially not-yet-retired instructions. If the address later turns out to be incorrect (due to a mispredicted branch or a cancelled instruction), the TLB lookup result is discarded. This speculative behaviour is architecturally transparent in the common case.

The question that becomes security-critical is: what happens when a speculative TLB access misses, and the processor speculatively initiates a page table walk for an address that the retiring instruction will never be permitted to access?

Speculative walks on present pages. For a PTE with P=1, a speculative walk installs a TLB entry that the eventual permission check at retirement may reject. If the instruction is cancelled or the access is a privilege violation, the processor squashes the result and does not deliver a fault. Any L1D cache fills triggered by the walk itself remain in the cache hierarchy regardless of whether the instruction retires.

Speculative walks on absent pages (P=0). The x86-64 architecture specification does not prohibit the processor from speculatively walking a page table entry that has P=0. The PTE with P=0 has an implementation-defined format in its non-present state; the physical address field bits [51:12] may contain data left by the OS. A processor that speculatively reads this field and issues a cache load to the resulting physical address has loaded data from an arbitrary memory location into the L1D cache, even though the page is marked not present. This data never reaches the architectural register file, but it is now in the cache and observable through timing side channels.

This mechanism is the basis of the Foreshadow / L1 Terminal Fault (L1TF) attack family, described in detail in Chapter 18. The key architectural gap is that the x86-64 specification, as of 2018, did not require that the processor suppress speculative walks on not-present PTEs. The hardware mitigation required a microcode update. The ARM64 specification explicitly leaves this behaviour implementation-defined; RISC-V software-managed TLBs are inherently immune since the walk executes in software where the OS can inspect the PTE before acting on it.

17.5 PTW Parallelism and Throughput

The throughput of the page table walker — measured in walks completed per cycle — is determined by three interacting factors: the number of concurrent walker slots, the latency of each walk, and the degree to which walks can be pipelined.

Walk latency dominates throughput. With two walker slots and a cold walk latency of 200 ns (800 cycles at 4 GHz), the maximum sustained PTW throughput is 2 walks per 800 cycles = 0.0025 walks per cycle. At a TLB miss rate of 0.01 misses per instruction and an IPC of 3, a processor needs 0.03 walks per cycle — twelve times more than two concurrent walkers can supply. This calculation, made concrete by Barr et al. [1], explains why large working set workloads with high TLB miss rates suffer catastrophic performance degradation.

Pipelining walks. The PTW pipeline is not a single-issue circuit. Each stage of the walk issues an independent memory read that goes into the cache hierarchy. These reads are issued sequentially within a single walk — stage N+1 cannot begin until the address from stage N is resolved — but two concurrent walks proceed independently through their respective stages. Walker 0 may be waiting for a DRAM fill at stage 2 while Walker 1 is completing an LLC hit at stage 4. This pipelining is what makes the two-walker design effective under real workloads with mixed LLC hit and DRAM miss rates.

Effect of huge pages on walk throughput. A 2 MB huge page PTE is found at the PDE level (three levels instead of four), reducing walk latency by one memory access and freeing the MSHR slot one stage earlier. The TLB coverage benefit of huge pages (512× more address space per entry) is documented in Chapters 4 and 9; the walker throughput benefit is a complementary and independent gain.

17.6 x86-64 vs ARM64 PTW Design Comparison

The x86-64 and ARM64 hardware page table walkers share the same high-level function but differ in implementation philosophy, fault delivery semantics, and the degree to which the walk interacts with the out-of-order pipeline.

x86-64: Microcode-assisted FSM. Intel's implementation uses a combination of a hardware finite-state machine and microcode routines stored in the processor's microcode ROM. The FSM handles the common-case walk (present pages, no reserved bit violations); exceptional cases — reserved bit set, access-rights violation, NX violation, certain PCID interactions — are handed off to microcode sequences that construct and deliver the precise exception. This design allows the common case to be fast while providing flexibility for complex fault semantics without additional hardware complexity.

ARM64: Pure hardware FSM. ARM's Translation Table Walker is implemented as a pure hardware state machine with no microcode involvement. The TTW specification is defined by the ARM Architecture Reference Manual's translation table walk algorithm (DDI 0487, §D8), and compliant implementations must produce identical results. This design is more predictable but less flexible; changes to walk semantics require silicon revision rather than a microcode patch.

Fault delivery timing. On x86-64, the #PF exception is delivered at instruction retirement. A speculative walk that detects a fault condition queues the fault; it is not raised until the faulting instruction reaches the head of the reorder buffer. On ARM64, a translation fault is also delivered as a synchronous exception at retirement, with the faulting virtual address in FAR_EL1 and the fault status in ESR_EL1.

PCID / ASID interaction. x86-64 PCID interacts with the PTW through CR3: the walker uses CR3[11:0] as the PCID to tag any TLB entries it installs. On ARM64, the ASID is stored in TTBR0_EL1[63:48] or TTBR1_EL1[63:48] and is similarly appended to TLB entries installed by the TTW.

17.7 RISC-V Software-Managed PTW

The base RISC-V privileged architecture specification deliberately omits a hardware page table walker. When a TLB miss occurs — detected by the hardware when satp.MODE is non-zero and the virtual address has no matching entry — the processor raises a page-fault exception and transfers control to the supervisor exception handler at the address in stvec. The handler is responsible for walking the page table in software, loading the appropriate entry, and executing SFENCE.VMA before returning.

Cost model. A software-managed TLB miss incurs the overhead of: trap entry (saving registers, switching to S-mode); computing the page table walk address from satp and the faulting virtual address in stval; loading each level of the page table tree with ordinary load instructions; writing the TLB entry using a privileged instruction; and executing SFENCE.VMA and returning. This sequence takes 100–1,000 cycles depending on whether the page table entries are in L1D cache, making RISC-V software TLB misses 10–100× more expensive than hardware-managed x86-64 or ARM64 misses for the same working set.

Advantages of the software approach. The software-managed model offers genuine engineering advantages for constrained implementations. It eliminates the area and power cost of a hardware PTW circuit, which matters for embedded RISC-V cores. It allows the OS to implement non-standard page table formats — hashed tables, inverted tables, compressed representations — that would require hardware changes on x86-64 or ARM64. And speculative PTW attacks (Chapter 18) are largely inapplicable because there is no hardware circuit performing speculative address translation.

Optional hardware walkers. Recognising the performance cost of pure software management, several RISC-V implementations add optional hardware page table walkers. The SiFive U74 core (used in the HiFive Unmatched board) includes a hardware PTW supporting Sv39 and Sv48. The T-Head XuanTie C906 (used in the Allwinner D1 SoC) similarly implements a hardware walker. These implementations conform to the RISC-V privileged specification — the walker installs TLB entries that satisfy the Sv39 translation algorithm — but are microarchitectural extensions not required by the ISA.

RISC-V H extension and nested walks. The RISC-V Hypervisor extension adds a second-stage translation (G-stage) that maps guest physical addresses to host physical addresses. In H-extension implementations with a hardware walker, a guest page walk that encounters a G-stage miss triggers a nested walk: the hardware must walk both the VS-stage and the G-stage page tables. This is analogous to Intel EPT and ARM Stage-2 nested translation, but implementation details are platform-specific.

17.8 Chapter Summary

The hardware page table walker is the circuit that converts a TLB miss into a TLB entry, and its microarchitectural properties determine the performance floor for any workload whose working set exceeds the TLB's coverage. This chapter has examined those properties in detail, from the pipeline stages of an individual walk to the system-level throughput limits imposed by concurrent walker capacity.

The x86-64 PTW proceeds through six sequential stages — TLB miss detection, CR3 load, and four page table level fetches — with each stage issuing a cache-coherent memory read to the L1D/L2/LLC hierarchy. Cold walk latency ranges from 12–50 cycles for LLC-resident page tables to 180–250 ns when all four levels miss to DRAM. The page walk cache (detailed in Chapters 3 and 4) eliminates upper-level fetches for temporally localised access patterns; its interaction with PTW-MSHR throughput is the key throughput-determining factor under real workloads.

Miss-Status Holding Registers enable concurrent page table walks by tracking multiple outstanding misses simultaneously. The PTW-MSHR coalescing mechanism ensures that multiple loads to the same 4 KB page share a single walk, while parallelism across independent MSHR entries allows two or more walks to proceed simultaneously through different pipeline stages. The ceiling of two concurrent walkers on Intel Haswell through current-generation cores imposes a hard throughput limit that becomes the binding constraint for workloads with high TLB miss rates and LLC-cold page tables.

Speculative page table walks — where the hardware initiates a walk for an address that may not ultimately be accessed — create a class of microarchitectural side channels in which the walk's cache side effects outlive the speculative instruction that triggered them. The mechanism by which a walk on a not-present PTE (P=0) can populate the L1D cache with data from an arbitrary physical address is the foundation of the Foreshadow / L1TF vulnerability family, examined in detail in Chapter 18.

The x86-64 microcode-assisted walker and the ARM64 pure-hardware FSM walker share similar pipeline depth and latency characteristics but differ in fault delivery semantics, speculative walk behaviour, and the flexibility with which walk semantics can be updated post-silicon. The x86-64 microcode path allows errata and security mitigations to be deployed as microcode updates without silicon changes; the ARM64 pure-FSM model provides stricter determinism and no microcode surface for post-silicon patching. The RISC-V software-managed approach represents a deliberate architectural trade-off — eliminating hardware walker complexity at the cost of 10–100× higher TLB miss latency — appropriate for embedded systems and custom memory subsystems but impractical for general-purpose server workloads without an optional hardware walker extension. Chapter 20 addresses the formal architectural consistency rules that govern when a walk's result must be considered stale, when explicit TLB invalidation is required, and what ordering guarantees each ISA provides around concurrent page table modifications.
